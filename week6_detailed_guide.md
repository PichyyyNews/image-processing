## บทที่ 1: สองวิธีมองภาพ — Spatial Domain กับ Frequency Domain

### 1.1 โดเมนเชิงพื้นที่ (Spatial Domain)

วิธีที่เราคุ้นเคยมากที่สุดในการประมวลผลภาพคือการมองภาพเป็น **ตารางตัวเลข** (Matrix) ที่แต่ละช่องเก็บค่าความสว่าง (Intensity) ของพิกเซลในตำแหน่ง (x, y) นั้น เราเรียกการทำงานในรูปแบบนี้ว่า **Spatial Domain** หรือโดเมนเชิงพื้นที่

ตัวอย่างเช่น ภาพขาวดำขนาด 512×512 พิกเซล ก็คือตารางตัวเลขขนาด 512×512 โดยแต่ละเซลล์มีค่าตั้งแต่ 0 (สีดำสนิท) ถึง 255 (สีขาวสนิท)

การประมวลผลใน Spatial Domain ทำได้โดยตรงและเข้าใจง่าย เช่น:
- ปรับความสว่าง: `pixel_new = pixel_old + 50`
- กลับสี (Invert): `pixel_new = 255 - pixel_old`
- กรอง Gaussian Blur: คำนวณค่าพิกเซลใหม่จากค่าเฉลี่ยถ่วงน้ำหนักของเพื่อนบ้านรอบข้าง

### 1.2 ข้อจำกัดของ Spatial Domain

แม้ Spatial Domain จะใช้งานง่าย แต่มีปัญหาบางอย่างที่แก้ได้ยาก โดยเฉพาะเมื่อ **Noise** หรือ **รูปแบบสัญญาณรบกวน** ที่ต้องการกำจัดนั้น "กระจายอยู่ทั่วทั้งภาพ" เช่น:

- ลายเส้นแนวนอนที่เกิดจากสัญญาณรบกวนไฟฟ้า (เช่น ภาพถ่ายจากกล้อง CCTV รุ่นเก่า)
- รูปแบบซ้ำๆ ที่เกิดจากการส่งสัญญาณ

เพราะในภาพที่มีลายเส้นแนวนอนซ้ำๆ ถ้าเรานึกภาพออก เส้นเหล่านั้นมี "ความถี่" ที่ชัดเจน — เราสามารถไปจัดการกับ "ความถี่" นั้นโดยตรงใน Frequency Domain ได้ง่ายกว่าการพยายามลบเส้นในภาพทีละเส้น

### 1.3 โดเมนเชิงความถี่ (Frequency Domain)

แนวคิดของ Frequency Domain คือการมองว่า "ภาพ" ไม่ได้ถูกสร้างจากพิกเซลจุดๆ แต่ถูกสร้างจาก **คลื่นซ้ำรูปแบบต่างๆ ที่ทับซ้อนกัน**

ให้นึกภาพการดูเพลง: เราสามารถมองเพลงเป็นคลื่นเสียงในโดเมนเวลา (Time Domain — เปลี่ยนแปลงตามเวลา) หรือเราอาจใช้ FFT แปลงเพลงให้อยู่ในรูปของ "กราฟความถี่" (Frequency Domain — ว่าในเพลงนี้มีเสียงความถี่ไหนดังแค่ไหน) ซึ่งทำให้เราแยก "เสียงเบส" ออกจาก "เสียงแหลม" ได้อย่างชัดเจน

**ภาพดิจิทัล** ก็ใช้หลักเดียวกัน เราสามารถแปลงภาพให้อยู่ในโดเมนความถี่ เพื่อดูว่า "ความถี่ในการเปลี่ยนแปลงของพิกเซล" มีรูปแบบอย่างไร:

| ประเภทความถี่ | ความหมายในภาพ | ตัวอย่าง |
|---|---|---|
| **ความถี่ต่ำ (Low Frequency)** | บริเวณที่สีเปลี่ยนแปลงช้าๆ | ท้องฟ้า, ผนังสีเดียว, พื้นหลัง |
| **ความถี่สูง (High Frequency)** | บริเวณที่สีเปลี่ยนแปลงเร็วและฉับพลัน | ขอบวัตถุ, ข้อความ, รายละเอียดละเอียดอ่อน, Noise |

> 💡 **จุดสำคัญที่ต้องจำ:** ความถี่สูงไม่ได้หมายความว่าสีสว่าง และความถี่ต่ำไม่ได้หมายความว่าสีมืด! "ความถี่" ในที่นี้หมายถึง "ความถี่ในการเปลี่ยนแปลง" ของค่าพิกเซล ไม่ใช่ค่าสีสว่างหรือมืด

## บทที่ 2: Fourier Transform — กลไกแปลงโลก

### 2.1 ประวัติและความสำคัญ

**Jean-Baptiste Joseph Fourier** (ค.ศ. 1768–1830) นักคณิตศาสตร์และนักฟิสิกส์ชาวฝรั่งเศส ได้นำเสนอทฤษฎีว่า "ฟังก์ชันคาบ (Periodic function) ใดๆ ก็ตาม สามารถแทนได้ด้วยผลรวมของฟังก์ชัน Sine และ Cosine ที่มีความถี่ต่างๆ"

สิ่งที่ทฤษฎีนี้บอกเราคือ: **ไม่ว่าสัญญาณจะซับซ้อนแค่ไหน เราสามารถแยกย่อยมันออกเป็นชิ้นส่วนความถี่ธรรมดาๆ ได้เสมอ**

ตัวอย่างง่ายๆ: สมมติมีคลื่นรูปร่างแปลกๆ — Fourier บอกว่าเราสามารถประกอบคลื่นรูปร่างนั้นจากการนำคลื่น Sine ที่มีขนาดและความถี่ต่างกันมาบวกรวมกัน

### 2.2 จาก Fourier Transform สู่ Discrete Fourier Transform (DFT)

ในทางคณิตศาสตร์ **Fourier Transform (FT)** ทำงานกับฟังก์ชันที่ต่อเนื่อง (Continuous) แต่ข้อมูลดิจิทัล (รวมถึงภาพ) เป็นข้อมูลไม่ต่อเนื่อง (Discrete) คือมีค่าเฉพาะจุดที่เป็นพิกเซล เราจึงต้องใช้เวอร์ชันดัดแปลงที่เรียกว่า **Discrete Fourier Transform (DFT)**

และเนื่องจากภาพมี 2 มิติ (แกน x และแกน y) เราจึงต้องใช้ **2D Discrete Fourier Transform (2D DFT)**

สูตรทางคณิตศาสตร์ของ 2D DFT:

$$F(u, v) = \sum_{x=0}^{M-1} \sum_{y=0}^{N-1} f(x, y) \cdot e^{-j2\pi\left(\frac{ux}{M}+\frac{vy}{N}\right)}$$

โดยที่:
- $f(x, y)$ = ค่าพิกเซลในภาพต้นฉบับ (Spatial Domain)
- $F(u, v)$ = ค่าในโดเมนความถี่ (Frequency Domain) ซึ่งเป็นจำนวนเชิงซ้อน
- $M, N$ = ขนาดความกว้างและความสูงของภาพ
- $u, v$ = พิกัดในโดเมนความถี่

> 🔑 **ไม่ต้องกังวลกับสูตรนี้มากนัก!** OpenCV มีฟังก์ชัน `cv2.dft()` ที่คำนวณให้ทั้งหมดในบรรทัดเดียว สิ่งที่สำคัญกว่าคือต้องเข้าใจ "ความหมาย" ของผลลัพธ์ที่ได้

### 2.3 ผลลัพธ์ที่ได้จาก 2D DFT — Magnitude Spectrum

ผลลัพธ์จาก DFT คือตาราง **จำนวนเชิงซ้อน (Complex Numbers)** ซึ่งแต่ละตัวประกอบไปด้วย:
- **ส่วนจริง (Real Part):** แสดงองค์ประกอบของคลื่น Cosine
- **ส่วนจินตภาพ (Imaginary Part):** แสดงองค์ประกอบของคลื่น Sine

สำหรับการ **แสดงผลเป็นภาพ** เราคำนวณหา **Magnitude (ขนาด)** ของจำนวนเชิงซ้อนแต่ละตัว:

$$|F(u,v)| = \sqrt{\text{Real}^2 + \text{Imaginary}^2}$$

จากนั้นนำมาผ่านฟังก์ชัน log เพื่อปรับ scale ให้ตาเราสามารถมองเห็นความแตกต่างได้:

$$\text{Magnitude Spectrum} = 20 \times \log(|F(u,v)|)$$

**การอ่าน Magnitude Spectrum:**

เมื่อนำภาพมาแสดงผล (หลังจากทำ Shift ให้ความถี่ต่ำมาอยู่กลาง) จะพบว่า:
- 🔵 **จุดสว่างตรงกลาง** = ความถี่ต่ำที่สุด (DC Component) มีพลังงานสูงสุด — หมายความว่าความสว่างโดยรวมของภาพ
- 🔵 **เมื่อออกห่างจากกลาง** = ความถี่ยิ่งสูงขึ้น ซึ่งโดยปกติจะมืดลงเรื่อยๆ (เว้นแต่ภาพมีโครงสร้างความถี่สูงที่ชัดเจน)
- 🔵 **จุดสว่างที่ผิดปกติ** นอกเหนือจากกึ่งกลาง = อาจเป็นสัญญาณรบกวนหรือรูปแบบซ้ำๆ ในภาพ

### 2.4 Fast Fourier Transform (FFT) — รุ่นเร็วของ DFT

การคำนวณ DFT โดยตรงนั้นใช้เวลา $O(N^4)$ สำหรับภาพ N×N ซึ่งช้ามาก ในทางปฏิบัติเราใช้ **Fast Fourier Transform (FFT)** ซึ่งเป็นอัลกอริทึมที่ชาญฉลาดในการคำนวณ DFT ให้เร็วขึ้นเป็น $O(N^2 \log N)$ 

NumPy มี `np.fft.fft2()` และ `np.fft.fftshift()` สำหรับใช้งาน FFT ที่รวดเร็ว ส่วน OpenCV มี `cv2.dft()` ซึ่งก็เร็วเช่นกัน

### 2.5 การแปลงกลับ (Inverse DFT)

เมื่อเราทำการกรองในโดเมนความถี่เสร็จแล้ว เราต้องแปลงภาพกลับสู่ Spatial Domain ด้วย **Inverse DFT (IDFT):**

$$f(x, y) = \frac{1}{MN}\sum_{u=0}^{M-1} \sum_{v=0}^{N-1} F(u, v) \cdot e^{j2\pi\left(\frac{ux}{M}+\frac{vy}{N}\right)}$$

OpenCV ทำให้ง่ายด้วยฟังก์ชัน `cv2.idft()`

## บทที่ 3: การกรองในโดเมนความถี่ — เลือกความถี่ที่ต้องการ

### 3.1 หลักการพื้นฐาน

กระบวนการกรองภาพในโดเมนความถี่ มี 3 ขั้นตอนหลัก:

```
[ภาพต้นฉบับ] --DFT--> [Frequency Domain] --ใส่ Filter/Mask--> [ผล Filtered] --IDFT--> [ภาพผลลัพธ์]
```

Filter ที่ใส่เข้าไปเป็น **Mask** ขนาดเดียวกับภาพ โดยค่าในแต่ละตำแหน่งคือ:
- **1** = อนุญาตให้ความถี่นั้นผ่าน (เก็บไว้)
- **0** = ตัดความถี่นั้นทิ้ง
- ค่าระหว่าง 0-1 = ลดทอนความแรงของความถี่นั้น

### 3.2 Low-pass Filter (LPF) — กรองเอาความถี่ต่ำ

**หลักการ:** อนุญาตให้เฉพาะความถี่ต่ำ (บริเวณกึ่งกลาง Spectrum) ผ่าน ตัดความถี่สูงรอบนอกทิ้ง

**วิธีสร้าง:** วาดวงกลมสีขาวตรงกลาง Mask (รัศมี r พิกเซล) ส่วนรอบนอกเป็นสีดำ

**ผลลัพธ์:** เมื่อแปลงกลับสู่ Spatial Domain จะได้ภาพที่:
- **เบลอลง (Blurring)** — เพราะขอบและรายละเอียดเล็กๆ (ที่เป็นความถี่สูง) ถูกตัดออก
- **Noise ลดลง** — Noise มักมีความถี่สูง จึงถูกตัดออกไปด้วย
- ยิ่ง r เล็ก → ยิ่งเบลอมาก | ยิ่ง r ใหญ่ → เบลอน้อย

**การประยุกต์ใช้:**
- กำจัด Noise ในภาพ
- ทำให้ภาพดูนุ่มนวล เตรียมก่อนการประมวลผลอื่น
- เลียนแบบเอฟเฟกต์เบลอแบบ Gaussian

**เปรียบเทียบกับ Gaussian Blur ใน Spatial Domain:**
| เกณฑ์ | LPF ใน Frequency Domain | Gaussian Blur ใน Spatial Domain |
|---|---|---|
| ความยืดหยุ่น | สูงมาก — กำหนดความถี่ตัดได้ละเอียด | จำกัดด้วยขนาด Kernel |
| การตัดความถี่ | ตัดขาดอย่างชัดเจน (Ideal LPF) | ค่อยๆ ลดลงแบบ Gaussian |
| ผลข้างเคียง | อาจเกิดลาย Ringing รอบขอบวัตถุ | เนียนตา ไม่มี Ringing |
| ความเร็ว | ช้ากว่าสำหรับ Kernel เล็ก | เร็วกว่าในทางปฏิบัติ |

> 📝 **หมายเหตุ: Ringing Effect** — เป็นปัญหาของ Ideal LPF ที่ตัดความถี่อย่างกะทันหัน ทำให้เกิดลายคลื่นรอบขอบในภาพ แก้ไขได้โดยใช้ Filter ที่ค่อยๆ ลดลง (เช่น Butterworth Filter หรือ Gaussian Filter)

### 3.3 High-pass Filter (HPF) — กรองเอาความถี่สูง

**หลักการ:** บล็อกความถี่ต่ำ (ตรงกลาง) และปล่อยให้ความถี่สูง (รอบนอก) ผ่าน

**วิธีสร้าง:** วาดวงกลมสีดำตรงกลาง Mask ส่วนรอบนอกเป็นสีขาว — หรือง่ายกว่านั้น ใช้ `HPF_mask = 1 - LPF_mask`

**ผลลัพธ์:** เมื่อแปลงกลับสู่ Spatial Domain จะได้ภาพที่:
- เห็นแต่ **เส้นขอบ (Edges)** และรายละเอียดคมชัด
- พื้นที่เรียบๆ กลายเป็นสีดำ (เพราะถูกตัดออก)

**การประยุกต์ใช้:**
- ขับเน้นเส้นขอบ (Edge Enhancement / Sharpening)
- ตรวจจับรูปแบบความถี่สูงในภาพ

**เปรียบเทียบกับ Canny Edge ใน Spatial Domain:**
| เกณฑ์ | HPF ใน Frequency Domain | Canny Edge Detection |
|---|---|---|
| เป้าหมาย | เห็นทุกความถี่สูง รวมถึง Noise | เห็นเฉพาะ Edge ที่แท้จริง |
| ความแม่นยำ | อาจตอบสนองต่อ Noise ด้วย | ทำ Noise Suppression ก่อน |
| การควบคุม | ควบคุมด้วยรัศมีวงกลม | ควบคุมด้วย Threshold 2 ค่า |
| ผลลัพธ์ | ขอบอาจยังมีความหนา | ขอบบางและชัดเจน |

### 3.4 Band-pass Filter — กรองเฉพาะช่วงความถี่

เป็น Filter ขั้นสูงที่อนุญาตให้เฉพาะความถี่ในช่วงที่กำหนด (ไม่ต่ำเกินไปและไม่สูงเกินไป) ผ่าน ใช้เมื่อต้องการโฟกัสเฉพาะรายละเอียดระดับกลางๆ

## บทที่ 4: Image Inpainting — ศิลปะการซ่อมแซมภาพ

### 4.1 Inpainting คืออะไร?

**Image Inpainting** (มาจากคำว่า "paint in" — วาดเติมลงไป) คือกระบวนการ **ถมและเติมบริเวณที่เสียหายหรือหายไปในภาพ** โดยให้ผลลัพธ์ดูเป็นธรรมชาติที่สุด คล้ายกับที่บริเวณนั้นไม่เคยมีปัญหาเลย

ในทางศิลปะ การ Inpainting หมายถึงการที่นักบูรณะผลงานศิลปะใช้พู่กันเติมสีในบริเวณที่ภาพเก่าแตกหรือหลุดออก โดยพยายามให้ตรงกับบริเวณรอบข้าง

ในทางคอมพิวเตอร์ OpenCV ทำสิ่งเดียวกัน แต่ใช้อัลกอริทึมทางคณิตศาสตร์แทนพู่กัน

**การใช้งานในชีวิตจริง:**
- ✅ ลบลายน้ำหรือตราประทับออกจากภาพ
- ✅ ซ่อมแซมภาพถ่ายเก่าที่มีรอยฉีกขาด รอยพับ หรือรอยเปื้อน
- ✅ ลบวัตถุที่ไม่ต้องการออกจากภาพ (เช่น ลบคนออกจากพื้นหลัง)
- ✅ กำจัดรอยขีดข่วนในภาพยนตร์ฟิล์มเก่า

### 4.2 สิ่งที่จำเป็น — ภาพต้นฉบับ และ Mask

เพื่อบอกให้อัลกอริทึม Inpainting รู้ว่า "ควรซ่อมแซมบริเวณไหน" เราต้องเตรียม 2 สิ่ง:

1. **ภาพต้นฉบับที่มีปัญหา (Damaged Image):** ภาพที่มีลายน้ำ รอยเปื้อน หรือบริเวณที่เสียหาย
2. **Mask (หน้ากาก):** ภาพขาวดำขนาดเท่ากัน โดย:
   - **สีขาว (255):** บริเวณที่เสียหาย ต้องการให้อัลกอริทึมซ่อมแซม
   - **สีดำ (0):** บริเวณปกติ ไม่ต้องแตะต้อง

```
ภาพจริง (มีลายน้ำ) + Mask (ขาว = ที่ลายน้ำอยู่) → cv2.inpaint() → ภาพที่ลายน้ำหาย
```

### 4.3 อัลกอริทึม INPAINT_NS — Navier-Stokes Inpainting

**ต้นกำเนิด:** อัลกอริทึมนี้ถูกพัฒนาโดย Bertalmio, Sapiro, Caselles และ Ballester ในปี ค.ศ. 2000 โดยได้รับแรงบันดาลใจจาก **สมการ Navier-Stokes** ซึ่งเป็นสมการทางคณิตศาสตร์ที่ใช้อธิบายการไหลของของเหลว

**หลักการ:** อัลกอริทึมมองข้อมูล "ความสว่าง" ในภาพเหมือนเป็น "ความหนาแน่นของของเหลว" แล้วจำลองการ "ไหล" ของความสว่างจากบริเวณนอกเขตที่เสียหาย เข้าสู่ภายในเขตที่เสียหาย

ขั้นตอนหลักๆ คือ:
1. อัลกอริทึมระบุ **Isophotes** (เส้นที่มีค่าความสว่างเท่ากัน) บริเวณขอบของเขตที่เสียหาย
2. จากนั้นนำเส้น Isophotes เหล่านั้น **"ลาก" ต่อเข้าไป** ในเขตที่เสียหาย
3. ทำซ้ำกระบวนการนี้จนเขตที่เสียหายทั้งหมดถูกเติมเต็ม

**ข้อดี:**
- รักษาความต่อเนื่องของเส้นขอบได้ดีเยี่ยม
- เหมาะกับการซ่อมแซมบริเวณที่มีโครงสร้าง เช่น เส้น ขอบ หรือรูปทรงเรขาคณิต

**ข้อเสีย:**
- คำนวณช้ากว่า TELEA สำหรับพื้นที่เสียหายขนาดใหญ่
- อาจทำงานได้ไม่ดีนักในบริเวณที่มี Texture ซับซ้อน

### 4.4 อัลกอริทึม INPAINT_TELEA — Fast Marching Method

**ต้นกำเนิด:** พัฒนาโดย Alexandru Telea ในปี ค.ศ. 2004 โดยอาศัยเทคนิคที่เรียกว่า **Fast Marching Method (FMM)**

**หลักการ:** คิดว่าพื้นที่เสียหายเหมือนน้ำที่กำลัง "ถูกเทลงมาจากขอบ" โดยขอบด้านนอกสุด (ที่มีข้อมูลภาพปกติ) จะถูกถมก่อน จากนั้น "แนวหน้า" ของการถมจะค่อยๆ ขยับเข้าสู่ด้านใน:

1. อัลกอริทึมคำนวณ **ลำดับการถมพิกเซล** จากขอบนอกเข้าสู่ใน (ใช้ Priority Queue)
2. แต่ละพิกเซลที่ต้องถม คำนวณค่าจาก **ค่าเฉลี่ยถ่วงน้ำหนัก** ของพิกเซลที่ถมแล้วรอบข้าง โดย:
   - พิกเซลใกล้กว่า = น้ำหนักมากกว่า
   - พิกเซลที่มีทิศทาง Gradient ตรงกัน = น้ำหนักมากกว่า

**ข้อดี:**
- เร็วกว่า NS สำหรับพื้นที่เสียหายขนาดใหญ่
- ให้ผลลัพธ์ดูเนียนตา

**ข้อเสีย:**
- รักษาโครงสร้างเส้นขอบได้ด้อยกว่า NS เล็กน้อย

### 4.5 เปรียบเทียบ NS กับ TELEA อย่างละเอียด

| เกณฑ์การเปรียบเทียบ | INPAINT_NS (Navier-Stokes) | INPAINT_TELEA (Fast Marching) |
|---|---|---|
| **ความเร็ว** | ช้ากว่า | เร็วกว่า |
| **การรักษาเส้นขอบ** | ดีเยี่ยม | ดี แต่ด้อยกว่าเล็กน้อย |
| **Texture ซับซ้อน** | ผลลัพธ์อาจไม่เนียน | ผลลัพธ์เนียนกว่า |
| **พื้นที่เสียหายใหญ่** | ทำงานได้แต่ช้า | แนะนำให้ใช้ |
| **พื้นที่เสียหายเล็ก/เส้น** | แนะนำให้ใช้ | ทำงานได้ดีเช่นกัน |
| **ค่า inpaintRadius** | ต้องการค่าน้อยกว่า | ต้องการค่าน้อยกว่า |
| **ใช้กับรูปแบบ** | บริเวณที่มีเส้นขอบชัดเจน | บริเวณ Texture สม่ำเสมอ |

**หลักเลือกใช้ในทางปฏิบัติ:**
- ถ้าซ่อมรอยแตกของภาพที่มีเส้น → ใช้ **NS**
- ถ้าลบลายน้ำจากพื้นสีเรียบ หรือพื้นที่ใหญ่ → ใช้ **TELEA**
- เมื่อไม่แน่ใจ → ลองทั้งสองอย่างแล้วเปรียบเทียบด้วยสายตา

### 4.6 inpaintRadius — พารามิเตอร์สำคัญ

`inpaintRadius` กำหนดว่าอัลกอริทึมจะ "มองไกลแค่ไหน" เมื่อคำนวณค่าพิกเซลที่จะถมเข้าไป:

- **ค่าน้อย (1-3):** มองพิกเซลเพื่อนบ้านใกล้ๆ — เหมาะกับรอยเปื้อนขนาดเล็ก รอยขีดบาง
- **ค่าปานกลาง (5-10):** เหมาะกับรอยเปื้อนขนาดกลาง ลายน้ำหนา
- **ค่ามาก (> 10):** มองไกล — เหมาะกับบริเวณที่เสียหายขนาดใหญ่ แต่ใช้เวลานานขึ้น

> ⚠️ **ข้อควรระวัง:** ค่า inpaintRadius ที่ใหญ่เกินไปอาจทำให้ผลลัพธ์เบลอหรือเกิดการ "ซึม" ของสีจากบริเวณที่ไม่ต้องการ

## บทที่ 5: การบีบอัดภาพ (Image Compression) — ลดขนาดโดยไม่สูญเสียมากเกินไป

### 5.1 ทำไมต้องบีบอัด?

ภาพถ่ายขาวดำขนาด 1920×1080 พิกเซล (Full HD) ที่ไม่ได้บีบอัด ต้องการพื้นที่:
- **Grayscale:** 1920 × 1080 × 1 byte = **~2.07 MB**
- **สีเต็ม (RGB):** 1920 × 1080 × 3 bytes = **~6.22 MB**

ภาพ 1,000 รูปในกล้องดิจิทัลจะกินพื้นที่ **6+ GB** ถ้าไม่บีบอัด! นั่นคือเหตุผลที่เราต้องการ Compression

### 5.2 ประเภทของการบีบอัด

| ประเภท | คำอธิบาย | ตัวอย่าง Format |
|---|---|---|
| **Lossless (ไม่สูญเสียข้อมูล)** | แตกไฟล์แล้วได้ข้อมูลเหมือนเดิมทุก bit | PNG, BMP, TIFF |
| **Lossy (สูญเสียข้อมูลบางส่วน)** | บีบอัดได้มากกว่า แต่ข้อมูลบางส่วนหายไป | JPEG, WebP |

### 5.3 JPEG Compression — เจาะลึกหลักการ

JPEG (Joint Photographic Experts Group) เป็นมาตรฐานบีบอัดภาพที่ใช้งานแพร่หลายที่สุดในโลก โดยอาศัยหลักการ 3 ขั้นตอนสำคัญ:

**ขั้นที่ 1: Color Space Conversion (การแปลงระบบสี)**
แปลงจาก RGB เป็น **YCbCr** โดยแยกข้อมูลความสว่าง (Y) ออกจากข้อมูลสี (Cb, Cr)
- เหตุผล: ตาคนไวต่อความสว่าง (Y) มากกว่าสี (Cb, Cr)
- จึงบีบอัดข้อมูลสี Cb และ Cr มากกว่าข้อมูลความสว่าง Y

**ขั้นที่ 2: Discrete Cosine Transform (DCT)**
แบ่งภาพออกเป็น **บล็อกขนาด 8×8 พิกเซล** แล้วแปลงแต่ละบล็อกด้วย **Discrete Cosine Transform (DCT)** ซึ่งทำงานคล้ายกับ Fourier Transform แต่ใช้เฉพาะ Cosine:
- ผลลัพธ์คือตาราง 8×8 ของ "ความถี่" ตั้งแต่ต่ำสุด (มุมบนซ้าย) ถึงสูงสุด (มุมล่างขวา)
- ข้อมูลส่วนใหญ่ของบล็อกจะ "กองอยู่ที่มุมบนซ้าย" (ความถี่ต่ำ)

**ขั้นที่ 3: Quantization (การปัดเศษเชิงคุณภาพ)**
นำค่าความถี่ที่ได้ มาหารด้วย **Quantization Table** แล้วปัดเป็นจำนวนเต็ม
- ค่าในตาราง Quantization ของความถี่สูง → มีค่ามาก → ทำให้ค่าความถี่สูงปัดเป็น 0
- นี่คือจุดที่ "ข้อมูลสูญหาย" ในการบีบอัด JPEG
- **Quality Factor** ที่เรากำหนด (0-100) จะส่งผลต่อ Quantization Table:
  - Quality 100 → ปัดเศษน้อย → ไฟล์ใหญ่ แต่ภาพคุณภาพดี
  - Quality 10 → ปัดเศษมาก → ไฟล์เล็ก แต่ภาพแตกเป็นบล็อก (JPEG Artifact)

### 5.4 JPEG vs PNG — เลือกใช้เมื่อไหร่?

| เกณฑ์ | JPEG | PNG |
|---|---|---|
| ประเภท Compression | Lossy | Lossless |
| ขนาดไฟล์ | เล็กกว่า (ปกติ 5-15× เล็กกว่า) | ใหญ่กว่า |
| คุณภาพ | ด้อยลงเล็กน้อย (ขึ้นกับ Quality) | เหมือนต้นฉบับทุก bit |
| Transparency | ไม่รองรับ | รองรับ (Alpha Channel) |
| เหมาะกับ | ภาพถ่าย, ภาพที่มีสีต่อเนื่อง | ภาพกราฟิก, Logo, ภาพที่ต้องการพื้นโปร่งใส |
| บันทึกซ้ำหลายครั้ง | คุณภาพแย่ลงทุกครั้ง | คุณภาพคงเดิมทุกครั้ง |

## บทที่ 6: ตัวชี้วัดคุณภาพภาพ — วัดความเสียหายด้วยตัวเลข

### 6.1 ทำไมต้องมีตัวชี้วัด?

เมื่อเราทำการบีบอัดหรือประมวลผลภาพ เราต้องการรู้ว่า "ภาพที่ได้มาแตกต่างจากต้นฉบับมากแค่ไหน?" ซึ่งการมองด้วยตาเปล่าอาจไม่เที่ยงตรงและบอกเป็นตัวเลขไม่ได้ ตัวชี้วัดทางคณิตศาสตร์จึงมีความสำคัญ

### 6.2 MSE (Mean Squared Error)

**MSE** คือค่าเฉลี่ยของความแตกต่างยกกำลังสองระหว่างพิกเซลของภาพสองรูป:

$$\text{MSE} = \frac{1}{MN}\sum_{x=0}^{M-1}\sum_{y=0}^{N-1}\left[I(x,y) - K(x,y)\right]^2$$

โดยที่ $I$ คือภาพต้นฉบับ, $K$ คือภาพที่ถูกดัดแปลง

**การตีความ:**
- MSE = 0 → ภาพสองรูปเหมือนกันทุก bit
- MSE สูง → ภาพมีความแตกต่างมาก (คุณภาพแย่ลง)
- MSE ไม่มีหน่วยที่ชัดเจน ทำให้ยากต่อการตีความในแต่ละบริบท

**ข้อจำกัดของ MSE:**
MSE อาจไม่สะท้อนการรับรู้ของสายตามนุษย์เสมอไป เช่น ภาพที่ Shift ตำแหน่ง 1 พิกเซลอาจมี MSE สูงมาก แต่ตามองแล้วแทบไม่ต่าง ในทางกลับกัน ภาพที่เบลอนิดหน่อยอาจมี MSE ต่ำ แต่ดูด้วยตาแล้วชัดว่าต่าง

### 6.3 PSNR (Peak Signal-to-Noise Ratio)

**PSNR** แก้ปัญหาของ MSE ที่ไม่มีหน่วย โดยเปรียบเทียบ MSE กับค่าพิกเซลสูงสุดที่เป็นไปได้:

$$\text{PSNR} = 10 \cdot \log_{10}\left(\frac{\text{MAX}^2}{\text{MSE}}\right) = 20 \cdot \log_{10}\left(\frac{\text{MAX}}{\sqrt{\text{MSE}}}\right)$$

โดยที่ MAX = ค่าพิกเซลสูงสุด = 255 (สำหรับภาพ 8-bit)

มีหน่วยเป็น **เดซิเบล (dB)**

**เกณฑ์การตีความ PSNR:**

| ช่วง PSNR | การตีความ |
|---|---|
| **∞ (Infinity)** | ภาพเหมือนต้นฉบับ 100% (MSE = 0) |
| **> 40 dB** | คุณภาพดีเยี่ยม ตาคนแทบแยกไม่ออกจากต้นฉบับ |
| **30 – 40 dB** | คุณภาพดี ยังอยู่ในเกณฑ์ที่ยอมรับได้สำหรับงานทั่วไป |
| **20 – 30 dB** | คุณภาพพอใช้ เริ่มเห็น Artifact ได้หากมองใกล้ๆ |
| **< 20 dB** | คุณภาพต่ำ เห็นความเสียหายชัดเจน |

### 6.4 ความสัมพันธ์ระหว่าง JPEG Quality และ PSNR

| JPEG Quality | ขนาดไฟล์ (เทียบกัน) | PSNR (ประมาณ) |
|---|---|---|
| 100 | ใหญ่ที่สุด (เทียบ PNG ได้) | > 45 dB |
| 80 | ~15% ของ Quality 100 | ~38-42 dB |
| 50 | ~5-8% ของ Quality 100 | ~33-38 dB |
| 20 | ~2-3% ของ Quality 100 | ~28-32 dB |
| 5 | เล็กมาก | < 25 dB |

| **High-pass Filter** | ตัดความถี่ต่ำ → เห็นเส้นขอบ ขับเน้นรายละเอียด |
| **Inpainting NS** | ซ่อมแซมโดยลาก Isophotes — เหมาะกับบริเวณที่มีเส้นขอบ |
| **Inpainting TELEA** | ซ่อมแซมจากขอบเข้าใน — เร็วกว่า เหมาะกับพื้นที่ใหญ่ |
| **JPEG Compression** | Lossy Compression ที่ทิ้งความถี่สูง ลดขนาดไฟล์อย่างมีประสิทธิภาพ |
| **MSE** | วัดความแตกต่างเฉลี่ยยกกำลังสอง ยิ่งสูงยิ่งแย่ |
| **PSNR** | วัดคุณภาพเป็น dB ยิ่งสูงยิ่งดี (> 30 dB = ยอมรับได้) |
